## Scope rule

Only files and settings explicitly listed in an accepted execution manifest may
be changed. Implementation is infrastructure-only and must not alter runtime,
Local AI, providers, governance authority, research/validation methodology,
published evidence, or LOCKED components.

## Required implementation manifest

Before execution, create a machine-checkable implementation manifest assigning
one unique identifier to every file and repository-setting action. The
manifest must record:

- exact path or setting;
- action;
- expected result;
- verification;
- rollback;
- protection classification; and
- whether the action changes GitHub state outside Git.

Any unlisted action is out of scope.

## Workflow implementation sequence

### Stage 1: Static controls

1. Create a dedicated branch from a synchronized accepted baseline.
2. Create CI003 and CI004.
3. Derive protected paths from authoritative records.
4. Test representative protected and ordinary paths locally.
5. Confirm the control detects but does not authorize protected changes.

### Stage 2: Required workflow

1. Create CI001 with minimal permissions.
2. Pin reviewed actions to full commit SHAs.
3. Implement PR, main-push, and manual triggers.
4. Implement base-aware whitespace checking.
5. Implement compilation without repository bytecode.
6. Implement canonical `unittest` regression on Python 3.14.
7. Record environment and dependency versions.
8. Avoid provider, Local AI, external runner, CLI, and research execution.

### Stage 3: Observational workflow

1. Create CI002 with minimal permissions.
2. Execute the existing coverage tool.
3. Produce and upload approved coverage artifacts for 14 days.
4. Add a pinned dependency scanner selected during implementation review.
5. Mark vulnerability policy descriptive during calibration.
6. Verify no source mutation, automatic dependency update, or public issue
   creation.

### Stage 4: Static configuration tests

Create CI005 to verify:

- exact event triggers;
- `contents: read` or stricter permissions;
- absence of write permissions and secrets;
- stable required job names;
- canonical regression command;
- Python 3.14 required environment;
- no `continue-on-error` or automatic retry for required jobs;
- no provider, Local AI, runtime CLI, research runner, deployment, release, or
  package-publication commands;
- full-SHA action pinning;
- artifact paths and retention;
- protected-path policy presence; and
- observational jobs not represented as compensating scores.

Tests should parse configuration safely. If an additional YAML parser is
required, its introduction needs explicit dependency treatment rather than an
undeclared environment assumption.

## Proposed GitHub settings

Repository settings are external state and require separate confirmation after
the workflow files have passed on a pull request.

| Setting ID | Proposed setting | Preconditions |
| --- | --- | --- |
| SET001 | Require pull request before merge to `main` | Required checks validated |
| SET002 | Require `scope`, `whitespace`, `compile`, and `regression` checks | Stable job names observed on GitHub |
| SET003 | Require branch current before merge | Required workflow reliability demonstrated |
| SET004 | Block force pushes to `main` | Confirm emergency recovery process |
| SET005 | Block deletion of `main` | Confirm repository administration ownership |
| SET006 | Require conversation resolution | Confirm normal review practice |

Do not enable or change repository settings in the workflow-file commit.
Settings changes need a separately recorded before/after state and rollback.

## Local validation before publication

Run:

```text
PYTHONDONTWRITEBYTECODE=1 python3 -m unittest tests.test_f11d_ci_configuration -v
PYTHONDONTWRITEBYTECODE=1 python3 -m unittest discover tests
pytest
git diff --check
```

Expected baseline:

- existing 226 tests remain passing;
- new static configuration tests add to, rather than replace, the baseline;
- `unittest` and `pytest` collect the same expanded suite; and
- no existing test is deleted, skipped, weakened, or rewritten merely to make
  CI pass.

## Workflow validation before merge

1. Push the implementation branch.
2. Open a pull request against `main`.
3. Verify every required and observational job starts under the intended event.
4. Preserve the first-run result, including failures.
5. If a failure reveals a design conflict, stop; do not weaken the gate.
6. Correct only accepted workflow defects in a new commit.
7. Verify all required jobs pass on the pull request.
8. Verify coverage and dependency reports are sanitized and downloadable.
9. Review the complete branch diff against the implementation manifest.
10. Obtain Product Owner acceptance before merge.

## Negative validation

Negative cases should be validated without committing defects to `main`.
Acceptable approaches include tests of local policy functions or temporary
branches/fixtures that are deleted after evidence is captured.

Required negative cases:

- trailing whitespace causes the whitespace check to fail;
- a representative protected path causes scope to fail and names the path;
- a failing test causes regression to fail;
- a syntax error causes compilation to fail;
- missing dependency installation causes an explicit diagnostic;
- an attempted write permission or secret reference fails static policy tests;
  and
- a prohibited execution command fails static policy tests.

Negative validation must not modify real P0 evidence or LOCKED files.

## Verification record

Implementation should create a documentation-only verification record after
workflow execution containing:

- baseline and implementation commit;
- workflow run URLs and IDs;
- action versions/SHAs;
- runner and Python identity;
- dependency manifest hash and installed versions;
- test counts and outcomes;
- compilation and whitespace outcomes;
- protected-path positive/negative outcomes;
- coverage scope and descriptive results;
- dependency-audit tool/result limitations;
- artifact names and expiry;
- changed-file list;
- P0/LOCKED boundary confirmation; and
- rollback readiness.

The verification record does not change workflow behavior and must not include
secrets or duplicate immutable research evidence.

## Rollback plan

### Before merge

- Stop at the first unapproved or unexplained result.
- Preserve workflow logs and branch commits.
- Do not modify required gates merely to obtain green status.
- Close the pull request or revert scoped workflow commits normally.

### After merge

- Revert the workflow/configuration commit through a reviewed pull request.
- Disable required branch checks only if they prevent the rollback and only
  through an explicitly recorded repository-setting action.
- Preserve the failed run and rollback evidence.
- Do not rewrite Git history.
- Confirm local regression still passes and protected surfaces remain
  unchanged.

### External settings

For each SET action, capture the prior GitHub setting and the exact restoration
operation before mutation. A settings rollback must not delete workflow or
review history.

## Stop conditions

Stop implementation and request review if:

- the accepted baseline differs locally or remotely;
- required actions are not available at immutable reviewed SHAs;
- Python 3.14 is unavailable on the selected runner;
- installing current requirements fails;
- test counts differ unexpectedly;
- CI requires provider credentials or network services beyond dependency/action
  retrieval;
- protected paths must be edited;
- research data would be uploaded;
- a workflow needs write permission;
- dependency scanning cannot produce a sanitized private result;
- branch-protection settings conflict with repository recovery needs; or
- any unlisted file or setting change appears.

## Commit and review structure

Recommended atomic boundaries:

1. protected-path policy and documentation;
2. required quality-gates workflow and static tests;
3. observational workflow and static tests;
4. verification evidence;
5. repository settings record, if separately authorized.

Do not squash implementation evidence if the accepted review policy requires
these boundaries to remain visible.
