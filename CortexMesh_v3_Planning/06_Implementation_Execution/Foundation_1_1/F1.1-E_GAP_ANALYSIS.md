## Purpose

Identify the minimum justified improvements for reproducible development,
testing, onboarding, and optional execution paths. This analysis does not
presume installable packaging and does not authorize a correction.

E1 remains the immutable current-state evidence record. E2 classifies those
findings and identifies the smallest defensible response without selecting
implementation mechanics.

## Classification vocabulary

| Classification | Meaning |
| --- | --- |
| Confirmed gap | Direct evidence shows that current documentation or environment control is insufficient for a stated or necessary repository workflow |
| Acceptable current constraint | A verified limitation that is consistent with the repository-first model and need not be removed now |
| Future design question | Evidence identifies a decision, but does not yet justify one answer |
| Unsupported claim requiring correction | Current presentation is broader or more definite than verified evidence |
| Deliberately deferred capability | The capability is outside the current minimum reproducibility objective and should not be implemented through F1.1-E |

These categories are independent of severity. A confirmed gap can have a small
correction; an acceptable constraint can still require explicit documentation.

## Decision principles

1. Preserve repository-first operation.
2. Correct claims before expanding compatibility.
3. Reproduce the verified environment before creating a distribution.
4. Separate optional execution paths from the default test path.
5. Do not relocate mutable state, sealed reference code, research evidence, or
   governance records through environment work.
6. Prefer documentation and environment declarations where they resolve the
   evidenced problem.
7. Treat package installation as a future outcome requiring measurable need,
   not as a standard Python-project default.

## 1. Python-version declaration versus verified support

### Classification

**Unsupported claim requiring correction.**

### Evidence

- `README.md` presents a general “Python 3” badge.
- Both implemented workflows select CPython 3.14 on `ubuntu-latest`.
- F1.1-D validation passed on CPython 3.14.
- The current developer host ran CPython 3.14.5.
- No `requires-python`, version matrix, minimum-version test, or compatibility
  policy exists.

### Impact

A contributor can reasonably interpret “Python 3” as broad compatibility,
while the repository demonstrates only CPython 3.14. Setup failures on older
versions would be difficult to classify as defects or unsupported use.

### Smallest viable correction

Replace the broad presentation with a precise statement:

```text
Verified environment: CPython 3.14 on Ubuntu CI.
Other Python versions and operating systems are not currently certified.
```

The local macOS evidence may be described as an observed development
environment, not a support promise.

### Risk

Low documentation risk. There is a higher risk if the change is framed as a
minimum-version contract without testing that boundary.

### Disposition

| Question | Decision |
| --- | --- |
| Belongs in F1.1-E? | YES — claim alignment and environment documentation |
| Implement now? | NO — requires E3 design/manifest authorization |
| Deferred? | Compatibility expansion is deferred; claim correction is not |

## 2. Dependency categorization and reproducibility

### Classification

**Confirmed gap.**

### Evidence

- `requirements.txt` combines conditional OpenAI runtime support, pytest, and
  coverage.
- Requirements use lower bounds only.
- No lockfile, constraints file, hashes, or recorded local resolution exists.
- CI upgrades pip and records `pip freeze`, but does not enforce that resolved
  set on the next run.
- `pip-audit==2.10.1` is intentionally installed only in its CI job.
- The canonical `unittest` suite does not import pytest; active reference,
  replay, and RP-001 harness code is standard-library-only.

### Impact

Two fresh clones at different times can resolve different direct or transitive
versions. Users cannot tell which dependencies are needed for the default
runtime, testing, CI observation, or optional OpenAI execution. Installing the
single file installs more than the default fallback/reference paths require.

### Smallest viable correction

The minimum outcome is an explicit dependency-role and resolution policy:

- identify the dependencies required for default verification;
- identify optional OpenAI execution separately;
- identify test and observational tools separately;
- define how the project records a known-good CPython 3.14 resolution;
- define when and how that resolution is refreshed and verified.

This does not require package metadata. The exact artifact form—split
requirements, constraints, generated lock, or another repository-first
mechanism—is a **future design question** for E3 because platform portability,
hash behavior, transitive dependencies, and update ownership must be evaluated.

### Risk

- Overly rigid cross-platform pins could encode one runner rather than a valid
  environment contract.
- Splitting files without a documented install profile could make onboarding
  less reliable.
- Treating CI's incidental `pip freeze` as an authoritative lock could include
  tools or platform-specific transitives unintentionally.

### Disposition

| Question | Decision |
| --- | --- |
| Belongs in F1.1-E? | YES |
| Implement now? | NO |
| Deferred? | Artifact selection is deferred to E3 design; the reproducibility gap is not |

## 3. Environment setup documentation

### Classification

**Confirmed gap.**

### Evidence

- README provides clone, POSIX venv activation, requirements installation, and
  `unittest` execution.
- CI adds a specific Python version, pip upgrade, environment fingerprint,
  requirement hash, and resolved dependency record.
- Environment variables and optional service prerequisites are distributed
  across implementation and sealed Phase 1B records.
- No single current environment reference distinguishes default verification,
  live CLI, OpenAI, Local AI, Docker capability, and research profiles.

### Impact

The simplest test path is usable, but a new authorized developer must infer
which instructions are mandatory, which are optional, and which execution
paths mutate state or require services.

### Smallest viable correction

Provide one current repository-first environment guide with profiles:

1. clean-clone verification;
2. default fallback CLI;
3. optional OpenAI execution;
4. optional Local AI execution;
5. optional external capability execution;
6. research/replay entry points by reference only.

Each profile should state Python/OS evidence, commands, environment variables,
external services, filesystem effects, and what is not certified.

### Risk

Copying old evidence commands into current onboarding could convert historical
records into unsupported current promises. The guide must link to sealed
evidence without rewriting it.

### Disposition

| Question | Decision |
| --- | --- |
| Belongs in F1.1-E? | YES |
| Implement now? | NO |
| Deferred? | No; exact document structure awaits E3 |

## 4. Mutable-state initialization and protection

### Classification

**Confirmed gap** for onboarding and execution safety; **future design
question** for state relocation or abstraction.

### Evidence

- `main.py` calls `engine.runner.run_task`.
- `run_task` loads, updates, and saves `memory/memory.json`.
- `save_memory` atomically replaces the tracked file and may create
  `memory/memory.json.bak`.
- README describes the CLI but does not disclose this mutation.
- Canonical verification does not need a live CLI task.

### Impact

A new user following Getting Started may dirty the worktree and alter persistent
runtime state without knowing that execution is stateful. Packaging could
silently change the state location or write permissions.

### Smallest viable correction

- Make regression the first onboarding execution.
- Label the CLI as stateful before its command.
- Identify the tracked state and backup paths.
- Document a read-only pre-run check and the approved recovery/provenance
  process.
- Do not advise destructive reset commands or silently initialize alternate
  state.

### Risk

Changing the state location, initialization code, or persistence semantics
would affect runtime and protected surfaces. Documentation must not imply that
deleting or regenerating tracked memory is harmless.

### Disposition

| Question | Decision |
| --- | --- |
| Belongs in F1.1-E? | YES for documentation; NO for runtime state redesign |
| Implement now? | NO |
| Deferred? | State abstraction/relocation is deliberately deferred pending separate runtime authority |

## 5. Optional OpenAI setup

### Classification

**Confirmed onboarding gap** and **acceptable optional capability**.

### Evidence

- `engine/model_router.py` uses fallback output when neither
  `OPENAI_API_KEY` nor `OPENAI_ADMIN_KEY` is present.
- It imports the OpenAI SDK only on the keyed path.
- `engine/mode_manager.py` uses `OPENAI_API_KEY` availability in mode
  selection.
- `openai>=1.0.0` is declared for every requirements installation.
- README does not describe the fallback/API distinction or key-driven mode
  consequences.

### Impact

Users may believe an API key is mandatory, or may set a key without realizing
that behavior changes from fallback to external service use.

### Smallest viable correction

Document OpenAI as opt-in:

- default no-key behavior;
- accepted environment-variable names;
- external network/service effect;
- model identifier source;
- absence of a production-readiness claim;
- prohibition on committing secrets.

Dependency separation belongs with the reproducibility design, not this
documentation correction alone.

### Risk

Documentation must not expose secrets, invent account requirements, or imply
that the current model/API path is universally available.

### Disposition

| Question | Decision |
| --- | --- |
| Belongs in F1.1-E? | YES |
| Implement now? | NO |
| Deferred? | Provider behavior changes are deliberately deferred |

## 7. Docker and external-service assumptions

### Classification

**Confirmed gap** in provisioning traceability; **deliberately deferred
capability** for implementation.

### Evidence

- LOCKED `core/external_runner.py` invokes Docker to inspect, build, run, and
  kill capability containers.
- It assumes the base image name `cortexmesh-capability`.
- No current root Dockerfile, Compose definition, or base-image build
  instruction was found.
- Default regression does not require Docker.

### Impact

The external-capability path is not reproducible from a fresh clone using
current environment artifacts. A user cannot determine whether the base image
is externally supplied, historical, or intentionally unavailable.

### Smallest viable correction

Within F1.1-E, document only the verified boundary:

- Docker is optional and not needed for canonical tests;
- the capability path expects a local base image;
- current provisioning is not determined from the repository;
- the path must not be represented as fresh-clone ready.

Providing or changing an image, Dockerfile, network rule, or runner behavior is
not a documentation correction and is outside F1.1-E unless separately
authorized through the protected boundary.

### Risk

Inventing a base image could change isolation, dependency, network, and
security properties of a LOCKED execution boundary.

### Disposition

| Question | Decision |
| --- | --- |
| Belongs in F1.1-E? | YES for truthful environment status |
| Implement now? | NO |
| Deferred? | Image provisioning and runner changes are deliberately deferred |

## 8. Repository-root and import-path dependence

### Classification

**Acceptable current constraint.**

### Evidence

- Top-level modules use repository-absolute imports.
- Direct nested scripts sometimes insert the repository root into `sys.path`.
- Local AI and replay tests insert their implementation root into `sys.path`.
- Phase 2C tests load sealed files by path.
- The canonical commands execute from repository root.
- E1 concluded that repository-first operation is currently preferable.

### Impact

Commands run outside documented paths may fail or import unintended modules.
The layout is unsuitable for claiming conventional installed-package behavior.

### Smallest viable correction

Declare repository root as the supported working directory for setup, tests,
and general development commands. Document exceptions only where a verified
script deliberately resolves its own repository root.

Do not refactor imports merely to resemble an installable package.

### Risk

Import restructuring could change runtime behavior, break sealed reference
paths, or invalidate research reproducibility.

### Disposition

| Question | Decision |
| --- | --- |
| Belongs in F1.1-E? | YES as an environment constraint |
| Implement now? | NO |
| Deferred? | Import refactoring is deliberately deferred |

## 9. Clean-clone test and execution procedures

### Classification

**Confirmed gap.**

### Evidence

- README supplies the basic path but does not fingerprint the environment.
- CI supplies a more precise CPython 3.14/Ubuntu process.
- The canonical test path is non-mutating when bytecode is disabled.
- The live CLI is stateful.
- Optional service paths have separate prerequisites.

### Impact

A fresh-clone user lacks one procedure that proves repository identity,
environment identity, dependency resolution, regression success, and clean
status before choosing a stateful or external execution path.

### Smallest viable correction

Define a verification-first clean-clone sequence:

1. confirm repository revision and clean status;
2. create an isolated CPython 3.14 environment;
3. install the approved verification dependency profile;
4. record Python, pip, requirements hash, and resolved environment;
5. run canonical `unittest`;
6. optionally run `pytest` parity;
7. run whitespace/status checks;
8. stop before any stateful CLI, provider, research, or Docker execution.

This should mirror existing CI semantics without requiring CI artifacts or
modifying workflows.

### Risk

Calling a process reproducible without constraining its dependency resolution
would overstate the correction. The final claim must match the environment
control selected in E3.

### Disposition

| Question | Decision |
| --- | --- |
| Belongs in F1.1-E? | YES |
| Implement now? | NO |
| Deferred? | No; exact commands depend on dependency-policy design |

## 12. Threshold for future installable packaging

### Classification

**Future design question.**

### Current evidence

No PyPI, external distribution, stable import API, or installed CLI requirement
has been established. Repository layout, sealed evidence paths, and tracked
mutable state make full packaging structurally consequential.

### Evidence threshold

Installable packaging should not enter design merely because packaging metadata
is absent. It becomes justified only when evidence demonstrates at least one
durable consumer need and all applicable boundary evidence:

| Threshold | Required evidence |
| --- | --- |
| External or cross-repository consumption | Repeated need to import a defined API without cloning/operating the full repository |
| Stable public surface | Named modules/classes/functions with compatibility ownership and versioning rules |
| CLI installation value | Measured onboarding or operational benefit over `python3 main.py` |
| State separation | Approved semantics for mutable memory/config/data outside source installation, with migration and rollback |
| Distribution boundary | Explicit inclusion/exclusion policy protecting governance, research evidence, corpora, archives, secrets, and proprietary content |
| Import viability | Tests proving installed imports without path injection or repository-root dependence |
| Dependency profiles | Verified runtime/optional/development dependency model |
| Reproducibility | Build and installation artifacts reproducible under the declared environment |
| Legal authority | Proprietary distribution rights and terms approved after the outstanding legal review |
| Governance/protected impact | Evidence and authority for any affected LOCKED or sealed boundary |

### Decision rule

If the demonstrated benefit can be obtained through repository-first
environment documentation, dependency control, or a task wrapper without
relocating code/state, installable packaging remains unjustified.

### Smallest viable correction now

None. Preserve the threshold in planning records and reassess only when a
consumer need is evidenced.

### Disposition

| Question | Decision |
| --- | --- |
| Belongs in F1.1-E? | Threshold definition belongs in E2; package implementation does not |
| Implement now? | NO |
| Deferred? | YES |

## Minimum justified F1.1-E scope

The evidence supports a narrow future design scope:

1. align Python/platform claims with CPython 3.14 Ubuntu evidence;
2. define dependency roles and a repository-first reproducibility policy;
3. establish one verification-first environment/onboarding guide;
4. disclose mutable CLI state and safe execution boundaries;
5. document optional OpenAI and Local AI profiles;
6. state Docker/capability provisioning limitations truthfully;
7. declare repository-root execution as the current supported model.

The evidence does not support:

- installable package creation;
- PyPI or other external publication;
- source-tree or import restructuring;
- runtime state relocation;
- Docker image creation;
- provider installation or selection;
- cross-platform support claims;
- workflow changes;
- dependency upgrades;
- changes to governance, research, validation, or LOCKED surfaces.

## Risks requiring E3 treatment

| Risk | Required design response |
| --- | --- |
| Dependency control becomes platform-specific or stale | Define generation/update ownership, verification, rollback, and compatibility scope |
| Documentation accidentally authorizes state mutation | Separate verification from stateful execution and retain explicit stop points |
| Optional profiles become implied defaults | Make opt-in boundaries and external effects prominent |
| Historical evidence is rewritten as current instruction | Link to sealed evidence; create new current guidance |
| Packaging work enters through dependency cleanup | Explicitly prohibit package metadata and import restructuring in the execution manifest |
| Docker limitation becomes an invented image | Preserve NOT DETERMINED status and defer protected implementation |
| Compatibility claims outrun validation | Use verified/observed/untested vocabulary consistently |

## Protected and excluded surfaces

E2 does not authorize changes to:

- `requirements.txt` or installed dependencies;
- packaging, environment, container, or task-runner artifacts;
- GitHub workflows, Actions pins, settings, or secrets;
- runtime, memory behavior, imports, or source layout;
- Local AI, OpenAI, provider selection, or provider adapters;
- Docker execution or capability images;
- governance authority or records;
- Phase 2C reference implementation;
- Phase 3A replay/validation assets;
- RP-001 protocols, harnesses, evidence, results, or observations;
- LOCKED components or the Lock Registry;
- licensing or external distribution.

## Conclusion

The minimum justified improvement is an explicit repository-first environment
contract, not an installable package. F1.1-E should correct the unsupported
Python claim, make dependency roles and resolution reproducible, provide a
verification-first onboarding path, and document stateful and optional
execution boundaries.

Repository-root imports are an acceptable current constraint. Cross-platform
support, capability-image provisioning, runtime state redesign, import
refactoring, and installable packaging remain deferred or require separate
evidence.

No implementation or design artifact is authorized by this analysis.
