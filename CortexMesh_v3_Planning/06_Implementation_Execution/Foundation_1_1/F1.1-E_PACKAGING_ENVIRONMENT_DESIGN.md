## Objective

Define the smallest coherent repository-first environment contract that lets a
competent engineer:

1. identify the verified environment;
2. create an isolated local environment;
3. install the correct dependency profile;
4. verify a clean clone before executing mutable or external paths;
5. use the stateful CLI knowingly;
6. configure optional OpenAI or Local AI execution;
7. distinguish repository-provided capabilities from operator-provisioned
   capabilities.

This is an environment and onboarding design. It is not an installable-package
design.

## Governing decision

> **CortexMesh remains a repository-first platform.**

The repository root remains the supported working directory. Source layout,
imports, mutable state, sealed reference components, replay assets, and
research evidence remain in place.

## Goals

- align support claims with demonstrated evidence;
- define dependency roles without adding general-purpose packaging machinery;
- make one environment guide authoritative;
- provide a verification-first clean-clone workflow;
- disclose persistent CLI state before execution;
- separate default, OpenAI, Local AI, and Docker capability profiles;
- record the limit of Docker capability provisioning;
- provide objective environment-contract tests;
- preserve the existing runtime and validation results.

## Non-goals

- `pyproject.toml`, `setup.py`, `setup.cfg`, wheels, editable installs, or
  console entry points;
- PyPI, container registry, or external distribution;
- import or source-tree refactoring;
- runtime state relocation, initialization redesign, or reset tooling;
- Docker image creation or `ExternalRunner` modification;
- provider installation, selection, or Local AI architecture changes;
- compatibility certification beyond the verified profile;
- dependency upgrades for their own sake;
- changes to governance, Phase 2C, Phase 3A, RP-001, or LOCKED components.

## Design principles

1. **Verified means continuously tested.**
2. **Expected means plausible but not certified.**
3. **Unsupported means no project assurance, not necessarily impossible.**
4. **Verification precedes stateful or external execution.**
5. **Dependency roles remain explicit.**
6. **A constraints record is evidence for a profile, not a universal lock.**
7. **Optional integrations remain opt-in.**
8. **Documentation cannot invent missing provisioning.**

## 1. Environment contract

### Authoritative support statement

The current support statement shall be:

```text
Verified environment:
CPython 3.14 on the Ubuntu environment exercised by CortexMesh CI.

Observed development environments:
Recorded macOS and Ubuntu executions exist, but observations outside the
verified CI profile do not create compatibility certification.

Other environments:
Other CPython 3.14 operating systems may be attempted. Other Python versions,
Windows, alternative Python implementations, and other platforms are not
currently certified.
```

The README badge and setup language must not imply generic Python 3
compatibility.

### Vocabulary contract

| Term | Required meaning |
| --- | --- |
| Verified | Required CI gates execute successfully on the stated profile |
| Observed | A recorded execution exists, but no continuing compatibility commitment follows |
| Expected | Architecture gives no known reason for failure, but the profile is not certified |
| Unsupported / not certified | The project makes no compatibility assurance |

“Supported” should be avoided unless it is immediately qualified by the
verified profile.

### Broader environment attempts

Broader environments may be attempted for development. Results must be
reported as observations. They must not alter the verified contract without a
separate compatibility design, repeatable tests, and Product Owner acceptance.

## 2. Dependency contract

### Dependency roles

| Role | Current dependency | Contract |
| --- | --- | --- |
| Default application/reference runtime | Python standard library | No third-party dependency is required for fallback CLI, Phase 2C reference logic, Phase 3A replay model, or RP-001 harness code inspected in E1 |
| Optional OpenAI runtime | `openai` | Installed only for the OpenAI profile; external service and credential required |
| Authoritative test | Python `unittest` | Standard-library canonical regression |
| Test parity / developer convenience | `pytest` | Development/test dependency; not application runtime |
| Observational coverage | `coverage` | Development/CI tooling; no release score |
| Descriptive dependency audit | `pip-audit==2.10.1` | CI-only pinned tool retained under F1.1-D; not application dependency |
| Optional Local AI | No Python package beyond repository code | Operator supplies Ollama or LM Studio service and model |
| Optional capability isolation | Docker CLI/daemon and base image | External/operator-provisioned; not a Python dependency |

### Candidate declaration layout

The implementation design uses repository requirements rather than package
metadata:

```text
requirements.txt
requirements-openai.txt
constraints/
    cpython314-ubuntu.txt
```

Contract:

- `requirements.txt` declares the direct verification/development tools needed
  by the documented default setup: pytest and coverage.
- `requirements-openai.txt` declares the optional OpenAI SDK and may include
  `-r requirements.txt` only if the selected implementation intentionally
  creates a combined developer profile. The manifest must fix one behavior.
- `constraints/cpython314-ubuntu.txt` records exact versions resolved and
  verified for the CPython 3.14 Ubuntu profile, including applicable
  transitives.
- `pip-audit` remains CI-local at its F1.1-D pin and is not moved into runtime
  or developer requirements.
- Local AI and Docker remain external prerequisites, not pip dependencies.

No empty runtime requirements file is justified: the default runtime's
standard-library status is clearer in the environment guide.

### Resolution policy

1. Human-maintained requirements express direct dependency intent and retain
   justified lower bounds.
2. The verified-profile constraints file records a known-good exact
   resolution, generated in a clean CPython 3.14 Ubuntu environment.
3. CI and the verified clean-clone procedure install direct requirements under
   that constraints file.
4. Optional OpenAI verification, if performed, uses the same profile-specific
   constraints only when all its transitives are represented.
5. A constraints update is an explicit maintenance change:
   - record previous and new resolution;
   - run dependency audit descriptively;
   - run 250/250 regression;
   - inspect protected identities;
   - use normal review and rollback.
6. Automated or floating dependency updates are not introduced by F1.1-E.

### Pinning decision

| Mechanism | Decision | Reason |
| --- | --- | --- |
| Lower bounds alone | Insufficient for verified-profile reproduction | Resolution can drift |
| Exact direct pins in every requirements file | Rejected as the only mechanism | Conflates dependency intent and one verified resolution |
| Profile-specific exact constraints | Selected | Separates declared intent from the tested resolution |
| Universal cross-platform lock | Not selected | Only Ubuntu/CPython 3.14 is verified |
| Hash-locked installation | Deferred design question | Platform artifact coverage and maintenance process are not yet evidenced |
| Package manager migration | Rejected | No measured need |

The constraints file must identify its generation environment and must not be
described as a universal lockfile.

## 3. Authoritative environment guide

### Candidate artifact

Create root-level `ENVIRONMENT.md` as the single current environment contract.
README remains the short entry point and links to it.

### Required sections

1. verified/observed/unsupported environment statement;
2. dependency profiles and file roles;
3. fresh-clone verification;
4. repository-root execution constraint;
5. stateful CLI warning and safe-use boundary;
6. optional OpenAI profile;
7. optional Local AI profile;
8. optional Docker capability limitation;
9. environment variables;
10. troubleshooting and explicit non-certifications.

Historical evidence remains linked rather than copied into a new claim.

## 4. Verification-first onboarding

### Authoritative sequence

```text
authorized clone
    ↓
confirm repository root, revision, and clean status
    ↓
verify CPython 3.14
    ↓
create and activate an isolated environment
    ↓
install the default verification profile under verified constraints
    ↓
record Python, pip, requirements/constraints hashes, and resolved dependencies
    ↓
run canonical unittest regression
    ↓
optionally run pytest parity
    ↓
run whitespace and clean-status checks
    ↓
read the stateful CLI warning
    ↓
choose default CLI or an explicitly configured optional path
```

### Candidate verified-profile commands

The exact commands must be fixed in the implementation manifest, but the
contract is:

```text
python3 --version
python3 -m venv .venv
source .venv/bin/activate
python -m pip install -r requirements.txt \
  -c constraints/cpython314-ubuntu.txt
PYTHONDONTWRITEBYTECODE=1 python -m unittest discover tests
PYTHONDONTWRITEBYTECODE=1 python -m pytest
git diff --check
git status --short
```

The guide must state that the constraints file is authoritative only for the
verified Ubuntu/CPython 3.14 profile. Non-verified environments may attempt a
direct-requirements installation but cannot claim equivalent reproduction.

### Clean-status expectation

Before optional execution, only expected local environment artifacts ignored
by `.gitignore` may exist. Canonical verification must not modify tracked
runtime memory, research evidence, validation assets, or governance records.

## 5. Stateful CLI contract

### Mutating command

```text
python3 main.py "<task>"
```

This command calls `engine.runner.run_task`, which loads and persists
`memory/memory.json`. A changed state may also produce
`memory/memory.json.bak`.

### Fresh-clone behavior

- The repository includes an existing tracked `memory/memory.json`.
- If the file is absent, current runtime code creates default state.
- F1.1-E does not redefine whether absence is an approved operating state.
- A normal CLI execution is expected to update state; a dirty worktree after a
  task is therefore possible and must not be described as test pollution.

### Safe-use instructions

Before a CLI task, the user should:

1. complete canonical verification;
2. inspect `git status --short`;
3. understand that tracked memory can change;
4. preserve any required state through an authorized Git/provenance process;
5. inspect the post-run diff before retaining or discarding it.

The guide must not prescribe `git reset --hard`, deletion, automatic backup,
silent reset, or regeneration as a universal recovery method.

### Prohibited design changes

F1.1-E must not:

- relocate state to a home/config directory;
- introduce a state environment variable;
- change initialization, backup, or persistence code;
- add a “stateless” CLI flag;
- alter memory schema;
- ignore the tracked primary memory file.

## 6. Optional OpenAI contract

### Activation

The OpenAI path is conditional:

- no `OPENAI_API_KEY` and no `OPENAI_ADMIN_KEY`: internal fallback behavior;
- configured key path: `engine.model_router` imports the OpenAI SDK and may
  make an external API request;
- `OPENAI_API_KEY` also affects mode availability through
  `engine.mode_manager`.

### Requirements

- optional OpenAI dependency profile installed;
- authorized credential supplied by environment;
- external network and service availability;
- current configured model remains `config/model.py` behavior and is not
  changed by F1.1-E.

### Failure behavior

- missing keys must remain a valid fallback condition;
- a configured keyed path without the SDK currently raises a clear runtime
  installation error;
- service/authentication/model errors remain external runtime failures and are
  not masked by environment documentation.

### Security

- never commit credentials or place them in examples;
- `.env` remains ignored, but F1.1-E does not introduce an `.env` loader;
- do not print, freeze, upload, or include secret values in evidence;
- environment fingerprints list variable names/status only, never values.

## 7. Optional Local AI contract

### Operator prerequisites

The operator supplies:

- a running Ollama or LM Studio service;
- a loaded/available model;
- endpoint reachability;
- sufficient local compute resources;
- the applicable `CORTEX_LOCAL_AI_*` configuration.

CortexMesh supplies provider-neutral repository adapters and manager logic; it
does not install the service or model.

### Configuration source

The authoritative guide lists:

- `CORTEX_LOCAL_AI_ENABLED`;
- `CORTEX_LOCAL_AI_PROVIDER`;
- `CORTEX_LOCAL_AI_PROVIDER_OPTIONS`;
- `CORTEX_LOCAL_AI_BASE_URL`;
- `CORTEX_LOCAL_AI_MODEL`;
- `CORTEX_LOCAL_AI_TIMEOUT`;
- `CORTEX_LOCAL_AI_TEMPERATURE`;
- `CORTEX_LOCAL_AI_MAX_TOKENS`.

Defaults must be derived from current code, not historical examples. Model
names from historical evidence must not become universal defaults.

### Service checks

The guide may point to existing provider health/verification interfaces and
tests. It must not trigger a live request during default setup or claim service
availability based on mocked regression tests.

### Unavailable service behavior

- disabled Local AI retains existing internal solver behavior;
- enabled but unavailable/misconfigured service may fail through current
  validation or request behavior;
- F1.1-E does not add fallback, retry, provider selection, or error-handling
  semantics.

### Docker relationship

Docker is not required by the SAFE Local AI bridge described in
`agents/local_ai_bridge.py`. Docker belongs to the separate external capability
boundary and must not be presented as a Local AI prerequisite.

## 8. Capability provisioning limitation

### Repository-provided evidence

The repository can verify that LOCKED `core/external_runner.py`:

- invokes Docker;
- expects a base image named `cortexmesh-capability`;
- constructs derived images for capability modules;
- applies existing isolation flags.

### Not repository-provisioned

Current repository evidence does not provide:

- an authoritative base-image Dockerfile;
- a build/pull procedure;
- an image digest or registry source;
- a fresh-clone capability execution guarantee.

These are intentional current limitations once documented. They are not
silently repaired in F1.1-E.

### Operator-supplied boundary

An operator may possess an externally provisioned image, but that fact does not
make the repository path reproducible or certified. The environment guide must
label capability execution unavailable from repository instructions unless a
separately authorized provisioning record is established.

## 9. Packaging threshold

Installable packaging remains deferred until evidence demonstrates one or more
durable needs and the applicable safety prerequisites.

### Need thresholds

- repeated reuse by external Python projects;
- a stable, owned public import surface;
- multiple entry points that materially benefit from installation;
- an external distribution requirement;
- measured onboarding or operational improvement over repository-first use.

### Safety prerequisites

- resolved mutable-state location and migration semantics;
- package-data inclusion/exclusion rules;
- protection of governance, research evidence, replay corpora, and archives;
- installed-import tests without repository path injection;
- dependency and optional-feature model;
- sustainable compatibility/versioning commitment;
- proprietary distribution terms approved after legal review;
- authority for any LOCKED or sealed boundary impact.

Meeting a need threshold permits assessment; it does not automatically
authorize packaging.

No source, import, memory, provider, research, validation, governance, or
LOCKED path is a candidate.

## 11. Risks and mitigations

| Risk | Mitigation |
| --- | --- |
| Constraints generated from an uncontrolled environment | Generate on clean Ubuntu/CPython 3.14 and record tool/environment details |
| Optional OpenAI transitives omitted | Validate optional install separately before claiming it |
| Existing CI pin baseline disturbed | Change commands only; preserve F1.1-D Action SHAs |
| Documentation and workflows diverge | Contract tests compare commands, versions, and paths |
| CLI warning understates mutation | Tests require explicit `memory/memory.json` disclosure before CLI instructions |
| Secrets leak through examples or logs | Placeholder-free credential rules and static prohibited-pattern tests |
| Docker limitation becomes implied support | Require explicit “not provisioned” and “not fresh-clone certified” language |
| Packaging enters opportunistically | Static tests prohibit packaging metadata and execution manifest prohibits it |
| E1/E2 evidence changes | Record hashes and fail scope verification on modification |

## 12. Decision boundaries

This design establishes what a later implementation may propose. It does not
authorize:

- dependency resolution or file edits;
- workflow edits;
- documentation edits outside these E3 artifacts;
- test creation;
- status changes;
- package or image creation;
- any runtime or protected-surface change.
