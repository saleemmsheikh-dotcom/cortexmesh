## 1. Repository classification

### Conclusion

**CONFIRMED — hybrid governed engineering and research platform with an
application-style live CLI and repository-contained orchestration frameworks.**

It is not currently an installable Python library or distribution:

- `README.md` calls CortexMesh an engineering and research platform for
  governed AI orchestration;
- `main.py` is an application-style command entry point;
- `orchestrator.py`, top-level component directories, and `engine/runner.py`
  constitute a live stateful orchestration application;
- Phase 2C contains an isolated deterministic reference orchestration
  framework;
- Phase 3A and RP-001 contain replay, validation, and research tooling;
- no `pyproject.toml`, `setup.py`, `setup.cfg`, distribution metadata, or
  console-script declaration exists.

## 5. Import and execution evidence

### Import model

- Top-level live components use absolute repository imports such as
  `from engine.runner import run_task`, `from agents.solver import SolverAgent`,
  and `from memory.memory_store import load_memory`.
- Some directly executable nested files insert the repository root into
  `sys.path`.
- The Local AI bridge inserts
  `CortexMesh_v3_Planning/06_Implementation_Execution/Phase1B` into `sys.path`
  and then imports `local_ai`.
- Phase 1B and Phase 3A tests also insert their implementation directory into
  `sys.path`.
- Several Phase 2C tests load implementation files with
  `importlib.util.spec_from_file_location`, reflecting their isolated planning
  location rather than a normally installed package.
- Internal modules within Phase 1B, Phase 2C, and Phase 3A use package-relative
  imports once their package root has been made importable.

This is a **CONFIRMED repository-layout import model**, not an installed
distribution import model.

### Repository-root and `PYTHONPATH` assumptions

The documented CLI works from the repository root. `main.py --help` also
worked when invoked by absolute path from `/tmp`, because the script directory
becomes importable. This does not establish that every test, audit utility, or
reference component is location-independent.

The canonical test command uses discovery from repository root. Test files
import top-level repository modules and some explicitly alter `sys.path`.
No installed-package test mode is defined.

### Direct scripts versus modules

The repository primarily documents direct-script execution (`python3
main.py`, RP-001 harness paths, and audit utilities) plus test-module execution
(`python -m unittest`). No `python -m cortexmesh`, console script, or unified
task command is declared.

## 7. Packaging suitability

### Current conclusion

**INFERRED — repository-first reproducibility is presently the better-aligned
primary model. Limited installable packaging could offer measurable developer
value, but full packaging would not be a metadata-only change under the current
layout.**

Potential measurable value of installable packaging is limited but real:

- explicit Python compatibility metadata;
- separated optional/runtime/development dependency groups;
- consistent import discovery without repeated `sys.path` manipulation;
- a stable CLI entry point;
- easier use of selected reusable components from outside the repository root.

However, current evidence shows structural constraints:

- live modules are spread across multiple top-level directories;
- `main.py` and runtime behavior assume repository-contained mutable data;
- the Local AI package lives under the planning/evidence tree;
- isolated Phase 2C/3A components intentionally live under sealed planning
  paths;
- tests and research harnesses rely on those exact paths;
- governance, research evidence, replay corpora, archives, and runtime code
  coexist in one repository but should not automatically become distribution
  payload.

Therefore:

- **CONFIRMED:** adding metadata alone would not prove a correct or useful
  installable distribution.
- **INFERRED:** a full installable package would require import-boundary,
  data-file, mutable-state, and inclusion/exclusion decisions, with possible
  structural changes.
- **CONFIRMED:** PyPI publication is not required by any current objective.
- **NOT DETERMINED:** whether a narrow internal package, editable development
  install, CLI wrapper, or environment-only improvement would yield the best
  value. That question belongs to later gap analysis/design, not E1.

## Assessment conclusion

CortexMesh currently operates coherently as a repository-first hybrid
engineering/research and orchestration platform. Its documented setup is enough
to create a POSIX virtual environment and run the canonical suite, while CI
defines a stronger but single-profile CPython 3.14/Ubuntu verification
environment.

The most material verified packaging/environment issue is not the absence of
PyPI publication. It is the absence of an explicit, reproducible environment
contract separating the live application, optional provider paths, quality
tooling, and standard-library-only reference/research assets. Installable
packaging may offer value, but the current repository structure means it would
require deliberate structural and state-boundary decisions rather than a
generic metadata addition.

No change is recommended or authorized by this assessment. Gap analysis and
design remain separate future decisions.
