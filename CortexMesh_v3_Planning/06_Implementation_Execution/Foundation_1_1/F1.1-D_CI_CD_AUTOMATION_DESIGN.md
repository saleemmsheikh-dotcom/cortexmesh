## Objective

Design an automated verification pipeline that reproduces CortexMesh's proven
manual quality gates on GitHub without changing application behavior,
governance authority, research methodology, validation certification, provider
behavior, or runtime architecture.

## Goals

- Run the complete regression suite consistently on proposed and merged changes.
- Detect compilation and whitespace failures before merge.
- make protected-surface changes explicit and fail closed.
- Preserve useful failure diagnostics without publishing sensitive or immutable
  research evidence.
- Produce descriptive coverage and dependency-risk information.
- Establish required checks suitable for later branch protection.
- Keep workflow permissions, dependencies, and operational complexity minimal.
- Make local and CI commands directly comparable.

## Non-goals

F1.1-D design does not:

- implement any workflow or repository setting;
- change production, runtime, Local AI, provider, or orchestration code;
- define a new supported-version promise;
- reformat or lint the existing codebase;
- introduce a packaging system or lock file;
- change test expectations or reduce the 226-test baseline;
- certify security, production readiness, or commercial readiness;
- execute RP-001 experiments or modify published evidence;
- create governance authority or automate Board decisions;
- authorize LOCKED or P0 changes; or
- deploy software, publish packages, or invoke providers.

## Design principles

1. **Manual equivalence:** required CI commands mirror commands already proven
   locally.
2. **Fail closed at protected boundaries:** CI may block an unauthorized
   protected-path change but may never authorize one.
3. **Independent gates:** regression, compilation, whitespace, and boundary
   checks report independently. No aggregate score compensates for failure.
4. **Descriptive observations:** coverage, duration, and initial dependency
   findings do not become implicit authority or quality scores.
5. **Least privilege:** workflows receive read-only repository permissions and
   no secrets by default.
6. **Immutable action references:** third-party and official actions are pinned
   to reviewed full commit SHAs in implementation.
7. **No experimental execution:** CI validates code and repository integrity;
   it does not rerun research data collection.
8. **No hidden mutation:** CI must not commit, push, create releases, update
   badges, or rewrite evidence.

## Proposed workflow architecture

### Workflow 1: Required quality gates

Proposed path:

```text
.github/workflows/quality-gates.yml
```

Triggers:

- pull requests targeting `main`;
- pushes to `main`; and
- manual `workflow_dispatch`.

Permissions:

```text
contents: read
```

No token write permission, environment secret, provider credential, Local AI
endpoint, or deployment permission is required.

Recommended jobs:

| Job | Role | Gate |
| --- | --- | --- |
| `scope` | Determine the correct comparison base and report changed protected paths | Required |
| `whitespace` | Run a base-aware `git diff --check` | Required |
| `compile` | Compile active Python source and tests without writing bytecode into the repository | Required |
| `regression` | Install declared dependencies and run canonical 226-test regression on CPython 3.14 | Required |

Job names must remain stable after branch protection references them.

### Workflow 2: Observational quality signals

Proposed path:

```text
.github/workflows/quality-observations.yml
```

Triggers:

- pull requests targeting `main`;
- pushes to `main`; and
- manual `workflow_dispatch`.

Recommended jobs:

| Job | Role | Initial policy |
| --- | --- | --- |
| `coverage` | Run the existing coverage audit and retain reports | Descriptive; command failure blocks, percentage does not |
| `dependency-audit` | Report known dependency vulnerabilities | Descriptive during calibration |

Separating observations prevents an uncalibrated coverage percentage or
dependency-tool finding from silently becoming a release policy. Promotion to
a required release gate requires evidence, a documented threshold/failure
policy, and Product Owner acceptance.

## Trigger and concurrency strategy

- Pull-request runs validate the proposed merge result.
- Push-to-main runs confirm the authoritative merged state.
- Manual dispatch supports controlled diagnosis without changing policy.
- Pull-request runs use a concurrency group keyed to workflow and pull request,
  cancelling superseded runs for the same pull request.
- Main-branch runs are not cancelled; every merged state retains a result.
- Scheduled execution is deferred until dependency-audit policy and operational
  ownership are established.

## Python and environment strategy

### Required environment

The initial required regression environment is:

```text
Runner: ubuntu-latest
Python: 3.14
```

This is the smallest environment already supported by recorded macOS and Ubuntu
evidence. The workflow must record runner OS, Python version, dependency
manifest hash, and installed dependency versions.

### Version matrix

A required multi-version matrix is not yet justified because CortexMesh has no
published Python support policy or package metadata. Additional Python versions
may be evaluated as non-required observations, but they must not be advertised
as supported until F1.1-E defines compatibility and environment policy.

### Dependency installation

The proposed sequence is:

1. check out the exact commit;
2. set up CPython 3.14;
3. upgrade `pip` within the ephemeral environment;
4. install `requirements.txt`;
5. record `python --version`, `pip --version`, and `pip freeze`; and
6. run the independent jobs.

Dependency caching may use the operating system, Python version, and
`requirements.txt` hash. A cache miss must not change correctness. Cache
contents are disposable and never certification evidence.

## Required verification behavior

### Scope and protected boundaries

Implementation should introduce a reviewed machine-readable path policy:

```text
.github/ci/protected-paths.txt
```

The scope job compares the pull-request or push diff with that policy. A match
fails closed and reports exact paths. It does not assess merit, apply an
approval label, or bypass GG-02. Authorized protected changes must follow their
existing process and a separately approved CI handling design.

The initial policy should cover, at minimum:

- governance authority and LOCKED records;
- runtime and Local AI implementation;
- provider implementation;
- certified replay corpora;
- Phase 3A validation certification;
- RP-001 published raw/analysis evidence; and
- Research Observations and Research Logbook records.

The file itself is a CI control, not a replacement for the authoritative LOCKED
registry or governance documents.

### Whitespace

The workflow must compute the correct base for each trigger:

- pull request: merge base against the target branch;
- push: before/after range supplied by the event, with a safe initial-commit
  fallback; and
- manual dispatch: selected commit against its first parent or an explicitly
  reported no-diff state.

It then runs `git diff --check` over that range. A shallow checkout must not make
the base unavailable; implementation should fetch sufficient history.

### Compilation

Compilation covers active Python source, tests, Phase 2C reference
orchestration, Phase 3A replay implementation, and active research harness
code. Compiled output must be directed outside the repository or suppressed.
Historical archived helpers are not promoted to active code by this check.

### Regression

Canonical command:

```text
PYTHONDONTWRITEBYTECODE=1 python -m unittest discover tests
```

Acceptance requires:

- exit status zero;
- 226 tests collected at the design baseline;
- no silent test-count reduction; and
- test-count changes reviewed as explicit test-suite changes rather than CI
  maintenance.

CI must not invoke `main.py`, Local AI services, providers, external runners,
or research execution harnesses as part of the ordinary regression gate.

## Coverage strategy

Coverage uses the existing command:

```text
PYTHONDONTWRITEBYTECODE=1 python tools/audit/audit_coverage.py
```

Initial outputs:

- console summary;
- machine-readable XML produced by an explicitly documented follow-up command;
- optional HTML report; and
- environment/dependency fingerprint.

Coverage percentages are descriptive. No repository-wide threshold is proposed
because the current tool intentionally measures a defined subset and historical
records distinguish component coverage from total coverage. A future threshold
must define scope, baseline, exceptions, and regression semantics.

## Dependency and security scanning

The design permits a minimal dependency audit in an isolated observational job.
The selected scanner and version must be pinned and reviewed during
implementation. Initial findings:

- are retained as diagnostics;
- do not modify dependencies automatically;
- do not open public issues containing sensitive data;
- do not claim that absence of findings proves security; and
- do not become a required gate until severity, exception, remediation, and
  false-positive policies are accepted.

Secret scanning, code scanning, Dependabot, and repository security settings
are separate GitHub controls requiring their own availability, privacy, and
operational review. They are not silently enabled by this design.

## Artifact and log retention

| Artifact | Retention | Constraints |
| --- | --- | --- |
| Regression logs | GitHub default | No credentials or provider responses |
| Environment fingerprint | 14 days | Versions and hashes only |
| Coverage XML/text | 14 days | Descriptive, not certification |
| Coverage HTML | 14 days, optional | No source outside repository checkout |
| Dependency-audit report | 14 days | Sanitized; private repository only |

CI must not upload replay corpora, RP-001 raw/analysis data, Local AI data,
memory stores, `.env` files, tokens, provider outputs, or Git bundles.

## Failure reporting

Every failed job should expose:

- stable job and step name;
- exact failing command;
- exit code;
- Python and runner identity;
- relevant changed paths;
- test count and failing test identifiers where applicable; and
- artifact link when safe.

Retries must not hide an initial failure. A rerun is a separate observation.
Flaky-test suppression, automatic retries, and `continue-on-error` are
prohibited for required gates.

## Branch protection recommendations

After successful workflow validation and separate repository-settings approval:

- require pull requests before merging to `main`;
- require stable checks for scope, whitespace, compilation, and regression;
- require the branch to be current before merge;
- prevent force pushes and branch deletion;
- preserve conversation resolution requirements if used;
- avoid naming observational jobs as required checks initially; and
- document any administrator bypass as an exceptional audited action.

Branch protection cannot substitute for GG-02, Product Owner acceptance,
research authorization, or LOCKED-component approval.

## Execution sequencing and performance

1. Run scope and whitespace first because they are fast and diagnostic.
2. Run compilation and dependency installation in parallel where practical.
3. Run regression as the authoritative behavioral gate.
4. Run coverage and dependency observations independently.
5. Upload only approved artifacts, including on diagnostic failure where safe.

The current regression completes quickly; optimization should favor clarity and
isolation over complex caching or job fan-out. Latency is operationally
descriptive and is not a quality score.

## Risks and mitigations

| Risk | Mitigation |
| --- | --- |
| Unpinned Python dependencies produce drift | Record installed versions; defer lock strategy to F1.1-E |
| `ubuntu-latest` runner changes | Record runner image; consider a pinned OS label after support review |
| Protected-path policy diverges from authority records | Treat it as fail-closed aid; review it against authoritative records |
| CI appears to authorize protected work | State explicitly that CI only detects and blocks |
| Coverage is mistaken for a quality score | Keep descriptive and scope-labelled |
| Dependency findings create noisy failures | Calibrate as observational before defining a gate |
| Action supply-chain compromise | Pin reviewed action SHAs and use least permissions |
| Secrets leak into logs/artifacts | Use no secrets; sanitize and tightly scope artifacts |
| Test count falls silently | Record baseline count and make reductions explicit |
| Flaky retries hide instability | No automatic retries on required jobs |
| Workflow proliferation obscures policy | Begin with two small workflows and stable job names |
| CI modifies protected evidence | Read-only checkout and no commit/write permissions |
